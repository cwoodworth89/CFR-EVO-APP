# Supabase Setup Guide: CFR Dispatch Integration

This document outlines the database schema, security policies, and real-time settings required to connect the **Raspberry Pi Locution Dispatch Agent** (Python backend) and the **CFR EVO Client** (React/Vite frontend) using **Supabase**.

---

## 📊 Database Schema: `live_calls`

The primary table `live_calls` handles storing the parsed dispatch metadata, geocoded coordinates/rings, raw transcripts for debugging, and verified ground-truth fields for the admin feedback loop.

### Complete DDL Migration Script

Execute the following SQL script in your Supabase project's **SQL Editor** to create the table, configure default values, establish indices, and enable Realtime subscriptions:

```sql
-- 1. Create the live_calls table
CREATE TABLE IF NOT EXISTS public.live_calls (
    -- Primary Keys & Identifiers
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    dispatch_id TEXT UNIQUE NOT NULL,                       -- E.g., DISP-2026-AE2798
    timestamp TIMESTAMPTZ DEFAULT TIMEZONE('utc', NOW()) NOT NULL,
    
    -- Metadata Parsed by STT Agent
    incident_type TEXT NOT NULL DEFAULT 'Unknown Incident',
    responding_units TEXT[] NOT NULL DEFAULT '{}',          -- Array of unit codes, e.g. {"E1", "L1"}
    
    -- Geocoding & Target Info (Option 2 Format)
    target JSONB NOT NULL DEFAULT '{
        "address": "Unknown Location",
        "lat": null,
        "lng": null,
        "rings": []
    }'::jsonb,
    
    -- Ground Truth, Performance & Quality Logging
    raw_transcript TEXT,                                     -- Original text from STT
    sanitized_transcript TEXT,                               -- Sanitized text from agent
    confidence_score NUMERIC(5,2) DEFAULT 0.0,               -- Fuzzy GIS lookup / STT confidence
    verify_location BOOLEAN DEFAULT FALSE,                   -- Flag indicating location needs validation
    origins TEXT[] DEFAULT '{}',                            -- Source audio file origins or markers
    
    -- Audio metadata
    audio_url TEXT,
    audio_duration NUMERIC(6,2),
    
    -- Corrections Entered via Admin Web Interface
    verified_transcript TEXT,                                -- Actual ground-truth transcript
    verified_address TEXT,                                   -- Ground-truth address
    verified_incident TEXT,                                  -- Ground-truth incident type
    verified_units TEXT[],                                   -- Ground-truth units
    feedback_submitted BOOLEAN DEFAULT FALSE                 -- Flag set once reviewed by admin
);

-- 2. Enable Realtime on this table (for instant React WebSocket updates)
ALTER publication supabase_realtime ADD TABLE public.live_calls;

-- 3. Create indices for performance
CREATE INDEX IF NOT EXISTS idx_live_calls_timestamp ON public.live_calls (timestamp DESC);
CREATE INDEX IF NOT EXISTS idx_live_calls_feedback ON public.live_calls (feedback_submitted, verify_location);
```

## 🔒 Row-Level Security (RLS) Configuration

Because dispatch information may contain private addresses and incident descriptions, we restrict access using Supabase RLS.

### 1. Enable RLS
By default, block all access until explicit policies are added:
```sql
ALTER TABLE public.live_calls ENABLE ROW LEVEL SECURITY;
```

### 2. Write Policy: Raspberry Pi Agent
The Raspberry Pi agent connects using the private `service_role` key. 
*   **No policy required**: The `service_role` key bypasses RLS automatically. Keep this key safe on the station's Raspberry Pi env files (`SUPABASE_SERVICE_ROLE_KEY`).

### 3. Read Policy: CFR EVO Client & Tasker
Browser clients and Tasker access the database using the public `anon` key. To secure the WebSocket and REST endpoints:
*   We require users to be authenticated via standard Supabase Auth (e.g. sharing a station login, or registering individual responder accounts).
*   Create a policy allowing authenticated users to read records:

```sql
CREATE POLICY "Allow authenticated users to read dispatches" 
ON public.live_calls
FOR SELECT 
TO authenticated 
USING (true);
```

### 4. Update Policy: Admin Interface
To allow authenticated admins (or stations users) to submit corrections/ground-truth updates:

```sql
CREATE POLICY "Allow authenticated users to submit corrections" 
ON public.live_calls
FOR UPDATE 
TO authenticated 
USING (true)
WITH CHECK (true);
```

## 📡 CFR EVO Real-Time Subscription (React Frontend)

In your React frontend application, you can listen to new dispatches in real-time using the Supabase JS Client:

```javascript
import { createClient } from '@supabase/supabase-js'

const supabaseUrl = 'https://your-project-ref.supabase.co'
const supabaseAnonKey = 'your-anon-key'

export const supabase = createClient(supabaseUrl, supabaseAnonKey)

// Subscribe to real-time dispatch updates
const dispatchSubscription = supabase
  .channel('live-dispatches')
  .on(
    'postgres_changes',
    {
      event: 'INSERT',
      schema: 'public',
      table: 'live_calls'
    },
    (payload) => {
      console.log('New Dispatch Received!', payload.new)
      // Trigger browser alarm, map zoom, and routing
    }
  )
  .subscribe()
```

## 📱 Tasker Auto-Navigation integration (Ntfy alternative)
The Python agent publishes updates to `ntfy.sh` which triggers Android Tasker navigation. If you want to use the database directly, Tasker can query Supabase:
1.  Use Tasker's HTTP Request action.
2.  Set URL to: `https://your-project-ref.supabase.co/rest/v1/live_calls?select=target&order=timestamp.desc&limit=1`
3.  Add Headers:
    *   `apikey: <your-anon-key>`
    *   `Authorization: Bearer <your-anon-key>`
4.  Parse the returned JSON payload and trigger the Google Maps Intent.
